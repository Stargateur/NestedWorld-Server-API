This document describe the protocol used with the NestedWorld’s realtime server.

The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT", "SHOULD", "SHOULD NOT", "RECOMMENDED", "MAY", and "OPTIONAL" in this document are to be interpreted as described in RFC 2119.

1. Message format

The protocol is composed of messages formatted as MessagePack [1] objects.

A message is MessagePack object containing message data in the fields.

All messages have some common fields:

Field name Field type Description

type

string

(required) The message type, with the reserved name result for "responses"

id

string

(optional) An optional request ID used in request to track responses.
If provided by a request, the response MUST include the same ID.
Example
# Client -> Server
{
    "id": "c791aeb9-d9bd-4845-acb1-787d9f1d6dff", (1)
    "type": "chat:join-channel", (2)
    "channel": "general"
}
# Server -> Client
{
    "id": "c791aeb9-d9bd-4845-acb1-787d9f1d6dff", (3)
    "type": "result",
    "result": "success",
    "users": []
}
1 An ID generated by the client to identify the request
2 The request type, here chat:join-channel for the example
3 The ID sent in the request by the client

2. States

A message state represents the context in which a message is sent (authenticated, in combat, etc.)

This state is defined by specifics message fields and/or a previous sent message (in which case the state is associated with the connection)

In case a message is sent without the required state associated or with invalid state data, some errors specific to the state can be generated.

A message modifying the state associated with the connection WILL be explicitely documented as it.

3. Special messages

Some common messages with not specific context related to them.

3.1. result

This message is sent in response to a request message (all other message types are requests) when the request need only a binary response (i.e. ok or error)

All requests are not always followed by a response.
Fields
Field name Field type Description

result

string

(required) The result type: success or error

In case of success, the other message fields contains the response data.

In case of error, the message contains special fields, and the other fields are the error details:

Error fields
Field name Field type Description

kind

string

(required) An identifier which identify the "kind" of error.

message

string

(required) A formatted error message which describe the error.
Success example
{
    "type": "result",
    "result": "success"
}
Error example
{
    "type": "result",
    "result": "error",
    "kind": "NotAuthenticated",
    "message": "You are not authenticated"
}

4. Global

Here are described all the structures and messages global to the server.

4.1. States

4.1.1. authenticated

A message with the state authenticated contains informations about a logged user context.

This state is associated with the connection, therefore no specifics message fields are expected and can be associated with the authenticate message.

Errors

  • NotAuthenticated: The connection doesn’t have the authenticated state associated.

4.2. Messages

4.2.1. authenticate

CLIENT-ONLY

This message is sent by the client to authenticate it self and associate the connection to an existing session, by its token (given by the HTTP API).

When successful, this message add the state authenticated to the connection’s associated states.
Request fields
Field name Field type Description

token

string

(required) A valid session token given by the HTTP API.
Errors

  • InvalidToken: The session token is invalid.

Example
{
    "type": "authenticate",
    "token": "mt11MwyHz0wO2Kz0nQlcl6T4hmWGzTkk"
}

5.2.1. chat:join-channel

CLIENT-ONLY

States: authenticated

This message is sent by the client to ask joining a named channel.

Request fields
Field name Field type Description

channel

string

(required) Channel name
Response fields
Field name Field type Description

users

array[chat:User]

(required) The list of users currently in the channel
Errors

  • InvalidChannel: Invalid channel (doesn’t exists or misnamed)

Field name Field type Description

reason

string

(required) nonexistent | incorrect-name

  • Unauthorized: User not authorized to join this channel

Example
{
    "type": "chat:join-channel",
    "token": "yrY9A3lRQd2ekRjNv8EWfYIGGOSmRlJm", (1)
    "channel": "general"
}
1 Needed by the authenticated state

5.2.2. chat:part-channel

CLIENT-ONLY

States: authenticated

This message is sent by the client to ask parting a named channel where he already is.

If a user part a channel he hasn’t joined before, no errors are generated.
Request fields
Field name Field type Description

channel

string

(required) Channel name
Errors

  • InvalidChannel: Invalid channel (doesn’t exists or misnamed)

Field name Field type Description

reason

string

(required) nonexistent | incorrect-name
Example
{
    "type": "chat:part-channel",
    "token": "CfGIz3roiUXqZr25PLRA79uTuIBz+zp7",
    "channel": "general"
}

5.2.3. chat:send-message

CLIENT-ONLY

States: authenticated

This message is sent by the client to send a message in a named channel where he already is.

If the user send a message to a channel he hasn’t joined, no errors are generated.
Request fields
Field name Field type Description

channel

string

(required) Channel name

message

string

(required) The message content to send
Errors

  • InvalidChannel: Invalid channel (doesn’t exists or misnamed)

Field name Field type Description

reason

string

(required) nonexistent | incorrect-name

  • MessageTooLong: Message too long

Field name Field type Description

max-message-length

integer

(required) Max message length authorized
Example
{
    "type": "chat:send-message",
    "channel": "general",
    "message": "Hello world!"
}
{
    "type": "chat:send-message",
    "channel": "general",
    "message": "roll",
    "me": true
}

6.3.4. combat:send-attack

CLIENT-ONLY

States: authenticated combat

This message is sent by the client to tell the server the current user’s monster attack.

The user may send this message even if he is not the first to move, as the order of attacks will be evaluated server-side.

But sending this message multiple times before the server applied the first sent attack will generate an error.

Request fields
Field name Field type Description

target

integer

(required) Target’s monster ID

attack

integer

(required) Attack ID to send
Errors

  • InvalidTarget: Nonexistent target or not a valid target (in case the user send his own monster?)

  • InvalidAttack: Nonexistent attack or not in the user’s monster attacks list.

  • AlreadySent: The user has sent an attack which hasn’t been processed yet.

6.3.5. combat:attack-received

SERVER-ONLY, NO RESPONSE NEEDED

States: combat

This message is sent by the server when it has processed an attack sent by one of the combat’s participants.

Fields
Field name Field type Description

attack

integer

(required) The attack ID sent

monster

combat:attack-received:Monster

(required) The attacker monster new informations

target

combat:attack-received:Monster

(required) The attacked monster new informations
combat:attack-received:Monster
Field name Field type Description

id

integer

(required) Monster ID

hp

integer

(required) The monster’s new HP value
Example
{

        "id": 2,

        "id": 3,

6.3.6. combat:monster-ko

SERVER-ONLY

States: combat

This message is sent by the server when one of the monsters participating in the combat is KO.

If the KO monster was in the client side and there is other monsters available in client’s team, then he MUST send back a combat:monster-ko:replace or combat:flee message as response.

If the KO monster was in the opponent side and the combat’s type is wild, then the client MUST send back a combat:monster-ko:capture message as response.

Otherwise, no response is required from the client.

Fields
Field name Field type Description

monster

integer

(required) The KO monster’s ID

6.3.7. combat:monster-ko:capture

CLIENT-ONLY

States: authenticated combat

Fields
Field name Field type Description

capture

boolean

(required) Does the client want to capture the KO monster?

name

string

(optional) The name of the monster the client want to capture, if it want to rename it. Can be nil if the client don’t want to rename it.
Errors

  • InvalidResult: Generated if the client sent the wrong result for the combat:monster-ko message.

6.3.8. combat:monster-ko:replace

CLIENT-ONLY

States: authenticated combat

Fields
Field name Field type Description

user_monster_id

integer

(required) The UserMonster ID the client want to replace its KO monster with.
Errors

  • InvalidResult: Generated if the client sent the wrong result for the combat:monster-ko message.

  • InvalidMonster: Generated if the client sent an invalid monster.

6.3.9. combat:monster-replaced

SERVER-ONLY, NO RESPONSE NEEDED

States: combat

This message is sent by the server to all combat’s participants to notify that the KO monster has been replaced.

Fields
Field name Field type Description

monster

combat:Monster

(required) The new monster informations

SERVER-ONLY, NO RESPONSE NEEDED

States: combat

This message is sent by the server to notify combat’s participants the combat has ended.

This message contains informations about the progress of the client’s monster.

string

(required) Combat’s result for the client: win or lose

stats

(required) Informations about the client’s monster progress.
combat:end:Stats
Field name Field type Description

id

integer

(required) The client’s monster ID

exp

integer

(required) Experience points earned

level

integer

(required) Level(s) earned
Example
{
    "type": "combat:end",
    "combat": 3,
    "status": "win",
    "stats": {
        "id": 2,
        "exp": 1245,
        "level": 1
    }
}
